<objective>
Create the Cloudflare Pages Function that serves as the alert API endpoint. It fetches NDMA SACHET RSS alerts, parses the XML, normalises alerts to a common schema, filters by state, caches results in Cloudflare KV with 15-minute TTL, and returns JSON. Gracefully degrades to empty array on any upstream failure.

Purpose: This is the backend data layer for the live alert banner. Without it, pages have no alert data.
Output: A working `/api/alerts?state={code}` endpoint deployable via Cloudflare Pages Functions.
</objective>

  <action>
    1. Install fast-xml-parser: `npm install fast-xml-parser`

    2. Create `wrangler.toml` at project root with:

    3. Create `functions/api/alerts.js` at project root (NOT inside dist/) implementing:

       **Entry point:** `export async function onRequestGet(context)` (Pages Functions convention)

       **Request handling:**
       - Parse `?state=` query param, uppercase it, default to "ALL"
       - Build cache key: `alerts:{STATE_CODE}` (e.g., `alerts:MH`, `alerts:ALL`)
       - Check KV cache first: `context.env.ALERTS_KV.get(cacheKey, { type: "json" })`
       - If cache HIT: return cached JSON with `X-Cache: HIT` header
       - If cache MISS: call `fetchAndNormaliseAlerts(stateCode)`, store in KV with `expirationTtl: 900` (15 min), return with `X-Cache: MISS` header
       - Always include `Content-Type: application/json` and `Access-Control-Allow-Origin: *` headers

       **SACHET RSS fetching (`fetchAndNormaliseAlerts`):**
       - Fetch `https://sachet.ndma.gov.in/cap_public_website/rss/rss_india.xml` with 8-second timeout via `AbortSignal.timeout(8000)`
       - Parse XML using fast-xml-parser's `XMLParser` with options: `{ ignoreAttributes: false, attributeNamePrefix: "@_" }`
       - Extract items from `rss.channel.item` (handle single item as array)
       - For each RSS item, extract:
         - `id`: from `guid` (text content or `#text` property)
         - `headline`: from `title` -- extract English portion (appears first before any regional text; split on common delimiters if multilingual)
         - `severity`: Default to "Unknown" from RSS metadata (RSS items don't carry severity directly). See CAP fetching below.
         - `event`: from `category`
         - `area`: parse from title or description if available
         - `source`: from `author` field (e.g., "controlroom@ndma.gov.in (Maharashtra-SDMA)" -- extract the parenthetical part)
         - `link`: from `link` (CAP XML URL)
         - `effective`: from `pubDate` (convert to ISO string)
         - `expires`: not available in RSS; set to null

       **State filtering:**
       - Build a state-name mapping from the `author` field. The author format is typically `email (StateName-SDMA)` or `email (IMD CityName)`.
       - Map state codes to SACHET state names for matching. Create an object mapping codes to names:
         ```
         MH -> "Maharashtra", KA -> "Karnataka", DL -> "Delhi", TN -> "Tamil Nadu", etc.
         ```
         Use the full list from data/states.json (all 36 states/UTs). The state name appears in the author field.
       - If stateCode is "ALL": return all alerts (no filtering)
       - If stateCode is a specific state: filter alerts where the author field contains the state name OR where the title/area mentions the state name. Also include alerts from "NDMA" (national-level alerts).

       **CAP XML severity enrichment (OPTIONAL, with strict limits):**
       - After filtering to a specific state, if the filtered set has 10 or fewer items, batch-fetch CAP XML for each item to get severity. Use `Promise.allSettled` with 5-second individual timeouts.
       - From CAP XML, extract: severity from `alert.info.severity` (pick the `en-IN` language info block), expires from `alert.info.expires`, headline from `alert.info.headline` (prefer CAP headline over RSS title as it's cleaner).
       - If CAP fetch fails for any item, keep RSS-derived data with severity "Unknown".
       - If there are more than 10 items, skip CAP fetching entirely and use RSS data as-is.
       - If stateCode is "ALL", skip CAP fetching (too many items).

       **Severity sorting:**
       - Sort alerts by severity descending using: Extreme=4, Severe=3, Moderate=2, Minor=1, Unknown=0
       - Cap at 20 alerts maximum per response.

       **Expiry filtering:**
       - If an alert has an `expires` field and the time has passed, exclude it from results.

       **Response format:**
       ```json
       {
         "alerts": [{ "id", "headline", "severity", "event", "area", "source", "link", "effective", "expires" }],
         "cached_at": "ISO timestamp",
         "state": "MH"
       }
       ```

       **Error handling:**
       - Wrap ALL external fetches in try/catch
       - On any failure, return: `{ "alerts": [], "error": "unavailable", "cached_at": "ISO timestamp", "state": "XX" }` with HTTP 200
       - Cache error responses too, but with shorter TTL: `expirationTtl: 300` (5 min) to retry sooner
       - Never return 500. Always 200.

       **CORS preflight:**
       - Also export `onRequestOptions` that returns 204 with CORS headers for preflight requests.
