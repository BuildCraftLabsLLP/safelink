## Summary

Phase 4 adds PWA installability, offline caching, and SEO metadata to a Python-built static site deployed on Cloudflare Pages. All work happens in three places: `build.py` (Python static site generator), Jinja2 templates in `templates/`, and new static files placed directly in `dist/` at build time.

The standard approach for this phase is: (1) write `manifest.json` and `sw.js` as static files output by `build.py`, (2) add hreflang, canonical, manifest link, and geo.region tags to `base.html`, (3) generate `sitemap.xml` and `robots.txt` in `build.py`, and (4) add inline SW registration and offline detector script to `base.html`. No npm packages or build tools beyond what already exists are needed. Everything is hand-written vanilla JavaScript.

The critical constraint throughout is the 15KB/20KB per-page size limit. Budget analysis shows hreflang (10 tags) + canonical + manifest link + SW inline registration adds approximately 1,200 bytes per page. The largest current page is 18,135 bytes (Malayalam flood guide), which leaves 1,143 bytes safety margin. The `about.json` content file needs updating with full mission/team/disclaimer content before build.

**Primary recommendation:** All PWA and SEO features are implemented as build-time output files (manifest.json, sw.js, sitemap.xml, robots.txt) generated by extending `build.py`, plus additions to `base.html`. No new Python packages needed. No npm packages needed. All service worker logic in vanilla JavaScript.

### Why No Workbox

Workbox is a library that abstracts service worker caching. This project:
- Has zero JavaScript bundler
- Uses inline-only scripts
- Has a hard 15KB/20KB page limit
- Has simple caching needs (HTML stale-while-revalidate + API network-first)

Hand-written vanilla SW is ~80 lines. Workbox would add hundreds of KB. Do not use Workbox.

### No New Python Packages

Sitemap generation in Python using only `xml.etree.ElementTree` (stdlib). No `sitemapper`, no `lxml`. The build already walks all pages via `build.py`'s page builders - generate sitemap URLs from the same data loops.

### Pattern 2: Service Worker - sw.js Structure

**What:** Single service worker file with three caching strategies.

**Strategies by URL pattern:**
1. Pre-cache on install: `/` and `/emergency/` (homepage + national emergency numbers)
2. `/api/alerts*` — network-first (live data preferred, fall back to KV-cached response stored in SW cache)
3. All other HTML requests — stale-while-revalidate with 24-hour freshness

**Cache versioning:** Use a version string in cache name. On `activate`, delete old caches.

```javascript
// Source: MDN Web Docs + web.dev caching strategies (MEDIUM confidence)
const CACHE_VERSION = 'v1';
const STATIC_CACHE = `safelink-static-${CACHE_VERSION}`;
const HTML_CACHE = `safelink-html-${CACHE_VERSION}`;
const API_CACHE = `safelink-api-${CACHE_VERSION}`;

const PRECACHE_URLS = ['/', '/emergency/'];
const HTML_CACHE_MAX_AGE_MS = 24 * 60 * 60 * 1000; // 24 hours

// INSTALL: pre-cache critical pages
self.addEventListener('install', event => {
  event.waitUntil(
    caches.open(STATIC_CACHE)
      .then(cache => cache.addAll(PRECACHE_URLS))
      .then(() => self.skipWaiting())
  );
});

// ACTIVATE: clean up old caches
self.addEventListener('activate', event => {
  event.waitUntil(
    caches.keys().then(cacheNames =>
      Promise.all(
        cacheNames
          .filter(name => name !== STATIC_CACHE && name !== HTML_CACHE && name !== API_CACHE)
          .map(name => caches.delete(name))
      )
    ).then(() => self.clients.claim())
  );
});

// FETCH: route requests to correct strategy
self.addEventListener('fetch', event => {
  const url = new URL(event.request.url);
  if (url.pathname.startsWith('/api/')) {
    event.respondWith(networkFirst(event.request));
  } else if (event.request.mode === 'navigate' || event.request.headers.get('accept').includes('text/html')) {
    event.respondWith(staleWhileRevalidate(event.request));
  }
  // Non-HTML requests (if any) fall through to browser default
});
```

**Cache freshness enforcement (HTML pages - 24 hours):**

```javascript
// Source: gomakethings.com cache expiry pattern (MEDIUM confidence)
async function staleWhileRevalidate(request) {
  const cache = await caches.open(HTML_CACHE);
  const cached = await cache.match(request);

  // Check cache freshness via custom header
  if (cached) {
    const fetchedOn = cached.headers.get('sw-fetched-on');
    const isFresh = fetchedOn && (parseFloat(fetchedOn) + HTML_CACHE_MAX_AGE_MS) > Date.now();

    // Revalidate in background
    const networkFetch = fetch(request).then(response => {
      if (response.ok) {
        const headers = new Headers(response.headers);
        headers.set('sw-fetched-on', String(Date.now()));
        const cloned = new Response(response.clone().body, { headers });
        cache.put(request, cloned);
      }
      return response;
    }).catch(() => null);

    if (isFresh) return cached; // Serve from cache, update in background
    // Stale but still useful: return cached immediately, background update already kicked off
    return cached;
  }

  // Not in cache: fetch from network and cache it
  return fetch(request).then(response => {
    if (response.ok) {
      const headers = new Headers(response.headers);
      headers.set('sw-fetched-on', String(Date.now()));
      const cloned = new Response(response.clone().body, { headers });
      cache.put(request, cloned);
    }
    return response;
  });
}
```

**Network-first for /api/alerts:**

```javascript
// Source: MDN PWA Caching guide (HIGH confidence)
async function networkFirst(request) {
  const cache = await caches.open(API_CACHE);
  try {
    const response = await fetch(request);
    if (response.ok) {
      cache.put(request, response.clone());
    }
    return response;
  } catch {
    const cached = await cache.match(request);
    return cached || Response.error();
  }
}
```

### Pattern 4: hreflang Tags in base.html

**What:** 10 `<link rel="alternate" hreflang="...">` tags in `<head>`, one per language plus one `x-default`. Plus a `<link rel="canonical">` pointing to the page's own canonical URL.

**Rules (verified from Google docs):**
- Every page links to ALL 10 language variants including itself
- Every page links to itself as canonical
- `x-default` points to the English version (no language prefix)
- All URLs must be absolute (full domain)

**Template variable needed:** `canonical_url` — the absolute URL of the current page for all 10 languages.

**Template approach:** Pass `canonical_base = "https://safelink.serverlord.in"` and `current_path` (already exists) to generate tags in `base.html`:

```html
<!-- Source: developers.google.com/search/docs/specialty/international/localized-versions (HIGH confidence) -->
<link rel="canonical" href="https://safelink.serverlord.in{{ current_path }}">
<link rel="alternate" hreflang="x-default" href="https://safelink.serverlord.in{{ current_path }}">
<link rel="alternate" hreflang="en" href="https://safelink.serverlord.in{{ current_path }}">
<link rel="alternate" hreflang="hi" href="https://safelink.serverlord.in/hi{{ current_path }}">
...
```

**IMPORTANT:** The canonical for English pages uses the path as-is (no `/en/` prefix per I18N-06). The canonical for `/hi/` pages includes the `/hi/` prefix. The `x-default` always points to the English version.

**Byte cost:** ~922 bytes per page (acceptable given budget analysis: largest page after addition = 19,337 bytes, under 20,480 limit with 1,143 bytes margin).

**Complexity note:** For non-English pages, `current_path` in the template is already the English path (e.g., `/state/maharashtra/`). The language prefix is in the URL structure, not in `current_path`. Verify this against the template context variables before implementing.

### Pattern 6: sitemap.xml Generation

**What:** Single `sitemap.xml` covering all pages. 10,580 URLs total, well under 50,000 URL limit and under ~1.8MB estimated size.

**Decision (Claude's discretion):** Single sitemap file (no sitemap index needed). Include all pages of all languages. Exclude: none.

**Rationale for including all language pages:** Each language URL is a distinct page with unique content. Google should index all 10 language variants.

**Format:** XML with only `<loc>` (required). Omit `<lastmod>`, `<priority>`, `<changefreq>` — Google ignores priority and changefreq, and lastmod is only useful if consistently accurate (hard to maintain per-page).

```xml
<?xml version="1.0" encoding="UTF-8"?>
<urlset xmlns="http://www.sitemaps.org/schemas/sitemap/0.9">
  <url><loc>https://safelink.serverlord.in/</loc></url>
  <url><loc>https://safelink.serverlord.in/state/andhra-pradesh/</loc></url>
  ...
</urlset>
```

**Python generation pattern (stdlib only):**
```python
# Source: Google sitemap documentation (HIGH confidence)
def generate_sitemap(data, languages):
    base = "https://safelink.serverlord.in"
    urls = []

    # For each language, generate all page URLs
    for lang in languages:
        prefix = f"/{lang}" if lang != "en" else ""
        urls.append(f"{base}{prefix}/")
        for state in data["states"]:
            urls.append(f"{base}{prefix}/state/{state['slug']}/")
        for district in data["districts"]:
            state = states_by_code[district["state_code"]]
            urls.append(f"{base}{prefix}/state/{state['slug']}/district/{district['slug']}/")
        # ... cities, guides, content

    # Write XML
    xml_lines = ['<?xml version="1.0" encoding="UTF-8"?>',
                 '<urlset xmlns="http://www.sitemaps.org/schemas/sitemap/0.9">']
    for url in urls:
        xml_lines.append(f'  <url><loc>{url}</loc></url>')
    xml_lines.append('</urlset>')

    with open(DIST_DIR / "sitemap.xml", "w", encoding="utf-8") as f:
        f.write("\n".join(xml_lines))
```

### Pattern 9: Icons (SVG-based PNG generation)

**What:** Two PNG icons (192x192 and 512x512) generated by build.py at build time, OR hand-crafted and placed in `dist/` directly (since they're static assets).

**Decision (Claude's discretion):** Generate using Python's `Pillow` library or create as SVG and rasterize — OR simpler: generate a simple shield/link SVG and embed it as base64 in a tiny Python script that writes PNG bytes.

**Simplest approach with no new libraries:** Write the icon data directly. A 192x192 PNG of a solid #FF9933 background with white text "SL" is ~500 bytes as a programmatically generated PNG. Use Python's `struct` + `zlib` to write a valid minimal PNG, or use the `pillow` library.

**Preferred approach given constraints:** Use Python `Pillow` (if available) or write a minimal valid PNG using stdlib. Alternatively, generate SVG icon and use browser's ability to render SVG (but Chrome Android requires raster PNG for home screen icons).

**Minimum requirement verified:** Chrome requires at least 192x192 and 512x512 PNG icons for installability and Lighthouse passing.

### Anti-Patterns to Avoid

- **Using Workbox:** Adds hundreds of KB to dist. Not compatible with inline-only approach.
- **Importing libraries in sw.js:** `importScripts()` would add external dependency. Write vanilla SW only.
- **Relative URLs in sitemap:** Google requires absolute URLs. Always prefix `https://safelink.serverlord.in`.
- **Missing self-referencing hreflang:** Every page must link to itself. Missing this causes Google to ignore entire hreflang cluster.
- **Cross-language canonicals:** Do not set canonical for `/hi/state/maharashtra/` pointing to `/state/maharashtra/`. Each language page is canonical to itself.
- **Caching sw.js with long TTL:** Cloudflare Pages by default sets `Cache-Control: max-age=3600` (DEPLOY-03). This means SW updates can take up to 1 hour to propagate. Not a blocking issue for launch, but worth noting.
- **SW scope outside root:** If sw.js is not at `/`, it can't control pages at `/`. Must be at `/sw.js` (in `dist/` root).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Sitemap XML formatting | Custom string builder | Python's `xml.etree.ElementTree` or simple list + join | Handles escaping; `<loc>` values with `&` need entity encoding |
| SW caching logic | Complex cache management | Simple async/await pattern (see Code Examples) | Workbox is overkill; the vanilla pattern is ~80 lines |
| Icon generation | Canvas/WebGL rendering | Python Pillow (if available) or pre-baked PNG bytes | PNG format is complex; hand-encoding wastes time |
| Offline detection | Custom ping/polling | `navigator.onLine` + `window.online`/`offline` events | Browser events are accurate, no polling needed |

**Key insight:** For this project, "don't hand-roll" mostly means "don't over-engineer." The entire SW and manifest can be hand-written in ~100 lines of vanilla JS and ~30 lines of JSON. No abstractions needed.

## Common Pitfalls

### Pitfall 1: Hreflang on Non-English Pages Gets Current_Path Wrong

**What goes wrong:** The `current_path` variable in the template context is the English path (e.g., `/state/maharashtra/`). For a Hindi page, the actual URL is `/hi/state/maharashtra/`, but `current_path` in the template is still `/state/maharashtra/`.

**Why it happens:** The build loop passes `current_path` as the page's content path without the language prefix. The language prefix is handled by the URL function and the output directory prefix.

**How to avoid:** In `base.html`, generate hreflang URLs by combining `current_lang`'s prefix + `current_path`. For English pages `current_lang = "en"`, so no prefix. For Hindi, prefix with `/hi`. This is the same logic `url()` function already uses.

**Warning signs:** All hreflang tags on Hindi pages point to English URLs. Check: does `/hi/state/maharashtra/` have `hreflang="hi"` pointing to `/hi/state/maharashtra/` or `/state/maharashtra/`?

**Template fix:**
```html
{%- set lang_prefixes = [("en",""),("hi","/hi"),("ta","/ta"),("te","/te"),("bn","/bn"),("mr","/mr"),("kn","/kn"),("ml","/ml"),("gu","/gu"),("pa","/pa")] -%}
<link rel="canonical" href="https://safelink.serverlord.in{% if current_lang != 'en' %}/{{ current_lang }}{% endif %}{{ current_path }}">
<link rel="alternate" hreflang="x-default" href="https://safelink.serverlord.in{{ current_path }}">
{% for code, prefix in lang_prefixes %}
<link rel="alternate" hreflang="{{ code }}" href="https://safelink.serverlord.in{{ prefix }}{{ current_path }}">
{% endfor %}
```

### Pitfall 2: Service Worker Scope Mismatch

**What goes wrong:** SW registered from `/state/maharashtra/` with `register('sw.js')` (relative path) registers the SW at scope `/state/maharashtra/` — it only controls pages under that path.

**Why it happens:** Relative path in `register()` resolves relative to the page calling it.

**How to avoid:** Always use absolute path: `navigator.serviceWorker.register('/sw.js')`. Confirmed: the scope will be `/` which covers all pages.

**Warning signs:** SW registered successfully but pages at `/hi/` or `/guide/` are not intercepted. Check scope in DevTools > Application > Service Workers.

### Pitfall 3: SW Install Precache Fails → SW Install Fails

**What goes wrong:** `event.waitUntil(cache.addAll(PRECACHE_URLS))` fails if any precache URL returns a non-200 response. The entire SW install fails and the SW is never activated.

**Why it happens:** `cache.addAll()` is all-or-nothing. If `/emergency/` page doesn't exist at SW install time (404), install fails.

**How to avoid:** Only precache URLs that definitely exist and return 200: `/` and one other verified page. Verify the national emergency numbers page URL before hardcoding it.

**Current codebase note:** There is no `/emergency/` page in the current dist. The national emergency numbers page appears to be on the homepage. Precache only `/` at install time, or add the specific emergency page URL after verifying it exists.

**Warning signs:** DevTools shows SW in "waiting to install" state; Chrome console shows `cache.addAll()` error.

### Pitfall 4: 15KB/20KB Size Limit Violations from hreflang

**What goes wrong:** Adding hreflang tags to large non-English pages pushes them over the 20KB limit. Build fails.

**Why it happens:** 10 hreflang tags + canonical = ~1,000-1,250 bytes depending on URL length. Pages near the limit will fail.

**How to avoid:** Budget analysis shows the current largest page (18,135 bytes) after additions will be ~19,337 bytes, leaving 1,143 bytes margin. Monitor build output warnings. If any page fails, the `build.py` size check will catch it.

**Current largest pages:**
- `/ml/guide/flood/index.html` — 18,135 bytes (closest to limit)
- `/ta/guide/flood/index.html` — 17,810 bytes
- These are guide pages with dense content

**Warning signs:** Build fails with `[FATAL] ... bytes (limit 20,480)` after adding hreflang to base.html.

### Pitfall 5: Cloudflare Cache-Control on sw.js

**What goes wrong:** `sw.js` is served with `Cache-Control: max-age=3600` (the project's DEPLOY-03 setting). Chrome re-checks the SW every 24h max, but HTTP cache can prevent detection of updates.

**Why it happens:** Cloudflare Pages sets `max-age=3600` on all static assets (DEPLOY-03). The SW file is treated as a regular static asset.

**How to avoid:** Add a Cloudflare Pages header rule via `_headers` file in `dist/` to set `Cache-Control: no-cache` on `/sw.js`. This is safe and cheap — the SW file is tiny.

**Implementation:** Create `dist/_headers` file:
```
/sw.js
  Cache-Control: no-cache
```

**Warning signs:** Deployed SW update doesn't propagate to users; old SW continues serving stale pages.

### Pitfall 6: About Page Content Is Incomplete

**What goes wrong:** The existing `data/content/about.json` has placeholder content. The CONTEXT.md requires: mission, team, data sources (SACHET/IMD/NDMA/state portals), disclaimer (numbers may be outdated), contact/error reporting. Current `about.json` has generic text that doesn't meet the "required before launch" bar.

**Why it happens:** about.json was created in an earlier phase with minimal content.

**How to avoid:** Update `about.json` with full required content sections before build.

**Missing sections in current about.json:**
- Team section (who built it: ServerLord / Atharva Kulkarni — already in footer)
- Explicit disclaimer text ("numbers may be outdated; verify via official sources during an emergency")
- Error reporting contact (GitHub link or email)
- SACHET/IMD explicitly listed as data sources (currently only mentions "official websites")

```python
# Source: Google sitemap XML format docs (HIGH confidence)
import xml.etree.ElementTree as ET

def generate_sitemap(data, languages, dist_dir):
    BASE_URL = "https://safelink.serverlord.in"
    urls = []

    states_by_code = {s["code"]: s for s in data["states"]}

    for lang in languages:
        prefix = f"/{lang}" if lang != "en" else ""
        urls.append(f"{BASE_URL}{prefix}/")
        for state in data["states"]:
            urls.append(f"{BASE_URL}{prefix}/state/{state['slug']}/")
        for district in data["districts"]:
            state = states_by_code.get(district["state_code"])
            if state:
                urls.append(f"{BASE_URL}{prefix}/state/{state['slug']}/district/{district['slug']}/")
        for city in data["cities"]:
            state = states_by_code.get(city["state_code"])
            if state:
                urls.append(f"{BASE_URL}{prefix}/state/{state['slug']}/city/{city['slug']}/")
        for guide_type in data["guides"]:
            urls.append(f"{BASE_URL}{prefix}/guide/{guide_type}/")
        for page_name in data["content"]:
            urls.append(f"{BASE_URL}{prefix}/{page_name}/")

    urlset = ET.Element("urlset", xmlns="http://www.sitemaps.org/schemas/sitemap/0.9")
    for u in urls:
        url_el = ET.SubElement(urlset, "url")
        ET.SubElement(url_el, "loc").text = u

    tree = ET.ElementTree(urlset)
    ET.indent(tree, space="  ")
    with open(dist_dir / "sitemap.xml", "wb") as f:
        tree.write(f, encoding="utf-8", xml_declaration=True)

    print(f"  Sitemap: {len(urls):,} URLs -> sitemap.xml")
```

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Workbox for service workers | Vanilla JS service workers viable for simple cases | Always an option | For this project's simple needs, no reason to add Workbox |
| Lighthouse PWA category score (0-100) | PWA audit is now pass/fail criteria | ~2023 Lighthouse 10/11 | "Score ≥ 90" target should be re-interpreted as "pass all PWA audit checks" |
| Service worker required for Chrome install | Service worker is NOT strictly required by Chrome install criteria | 2023 | SW is still needed for offline + is best practice; Chrome will still show install for SW-less sites with valid manifest |
| `priority` and `changefreq` in sitemaps | Google ignores both fields | Years ago | Omit them; they add bytes with no benefit |

**Important clarification on Lighthouse PWA score:** Lighthouse's PWA audit is pass/fail per check, not a 0-100 score. The "≥ 90" requirement in CONTEXT.md likely refers to a combination of PWA audit passing + Lighthouse Performance score ≥ 90. The project currently has no JavaScript bundles and sub-7KB pages, so Performance score should easily reach 90+ once SW is registered. The key PWA checks to pass: valid manifest (installable) + registered service worker that controls start_url.

## Open Questions

1. **National emergency numbers page URL**
   - What we know: The homepage (`/`) is precached on SW install. The spec says also precache "national emergency numbers page."
   - What's unclear: There is no standalone `/emergency/` page in the current dist. National numbers appear on the homepage (`/`). Is there a dedicated page at a specific URL?
   - Recommendation: Only precache `/` on SW install. If a dedicated emergency numbers page is created as part of this phase (possible based on shortcut pointing to it), precache that URL too.

2. **App shortcut URL for "National Emergency"**
   - What we know: CONTEXT.md says shortcuts: "homepage + national emergency numbers page"
   - What's unclear: What URL does the national emergency numbers shortcut point to? The current site has no `/emergency/` page.
   - Recommendation: Either point the second shortcut to `/` (same as start_url), or create a minimal `/emergency/index.html` page for this purpose. The shortcut `url` must be within the manifest scope.

3. **Icon generation approach**
   - What we know: Need 192x192 and 512x512 PNG. Current project has no image generation dependencies. Pillow is not in requirements.txt.
   - What's unclear: Should Pillow be added as a dependency, or should icons be hand-crafted and committed to the repo as static files?
   - Recommendation: Create the icons as static PNG files and copy them to `dist/` in `build.py`. This avoids adding Pillow as a runtime dependency. The icons are fixed assets, not generated from data.

4. **hreflang for `/emergency/` page if it's created**
   - What we know: All pages get hreflang. If a new `/emergency/` page is created for the shortcut, it needs its own 10-language hreflang set.
   - What's unclear: Whether this page will be a content page (handled by the existing `build_static_pages` function) or a special case.
   - Recommendation: If created, add it as a content page entry in `data/content/` so it gets built for all 10 languages automatically.

5. **`<link rel="manifest">` on non-English pages**
   - What we know: The manifest `start_url` is `/` (English homepage). Non-English pages at `/hi/`, `/ta/`, etc. would still link to the same `/manifest.json`.
   - What's unclear: Does linking to a manifest from a non-English page cause Lighthouse to consider those pages installable with a `/hi/` start?
   - Recommendation: Link to `/manifest.json` from all pages. The manifest's `start_url: "/"` means the installed app always opens the English homepage. This is correct per CONTEXT.md decision ("start_url: / (English homepage)").
